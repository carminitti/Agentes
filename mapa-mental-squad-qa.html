⚙️ Como Cada Executor Funciona

Clique para expandir
🌐
executor-browser
Playwright + TypeScript
▼
  • Gera Page Objects (POM) para cada página testada
  • Seletores semânticos: getByRole, getByLabel, getByText — sem IDs frágeis
  • Setup (beforeAll): cria massa de dados via API com request.newContext()
  • Testes: executam contra os dados criados no setup
  • Teardown (afterAll): apaga tudo que foi criado, mesmo se testes falharem
  • Cobre smoke, sanity, regressão, e2e e cross-browser
  • Também processa testes http de smoke/sanity/regressão/e2e
  • Auto-registro restrito: apenas em domínios de demonstração conhecidos (AutomationExercise, DemoQA, etc.) ou com ALLOW_AUTO_REGISTER=true
🔵 Tipos cobertos: smoke · sanity · regressão · e2e · cross-browser
🔌
executor-api
Python requests
▼
  • Extrai método, endpoint, payload e headers dos steps de teste
  • Setup: cria pré-condições via API
  • Testes: valida status code, campos do body e tempo de resposta
  • Teardown (finally): bloco finally garante limpeza mesmo em falhas
  • Suporta GET, POST, PUT, PATCH, DELETE, autenticação Bearer/Basic
🟢 Tipos cobertos: integração · http (integração)
📈
executor-performance
k6 ou fallback Python
▼
  • Se k6 disponível: gera script JS nativo e executa
  • Se k6 ausente: fallback Python imediato (threading para simular VUs) — sem tentar instalar
  • Perfis: performance (10VUs/30s) · carga (50VUs/60s) · stress (rampa) · soak (20VUs/10min)
  • Métricas coletadas: p50 / p95 / p99, error rate, throughput
🟡 Tipos cobertos: performance · carga · stress · soak
👁️
executor-visual
Playwright screenshots
▼
  • 1ª execução: captura e salva imagens de baseline + aviso obrigatório para validação visual
  • Execuções seguintes: compara screenshots com o baseline
  • Threshold de diferença: 2% — abaixo disso é aprovado
  • Oculta elementos dinâmicos (datas, animações) para evitar falsos positivos
🟣 Tipos cobertos: visual
♿
executor-acessibilidade
axe-core via Playwright
▼
  • Padrão: WCAG 2.1 AA
  • critical / serious → falha o teste
  • moderate / minor → reportado como warning
  • Analisa cada página ou componente especificado
  • Falhas conhecidas de demo: violações anotadas como known_demo_failure, "violação conhecida", "aceito pelo time", etc. → não bloqueiam deploy; detecção case-insensitive
  • Ambientes de demonstração: deploy_blocked: false se todas as violações failed forem conhecidas
🟣 Tipos cobertos: acessibilidade
🔒
executor-seguranca
Python (não invasivo)
▼
  • Autenticação: verifica 401 em rotas sem token
  • Autorização: verifica 403 em rotas sem permissão
  • Headers: HSTS, X-Frame-Options, CSP, X-Content-Type
  • CORS: valida via GET + OPTIONS preflight — detecta APIs que só respondem ao preflight
  • Endpoints sensíveis: /admin, /env, /debug, /.git, etc.
⚠️ Nunca executa: SQL injection · XSS · fuzzing · força bruta — apenas verificações passivas e seguras.
🗄️
executor-banco
Python + drivers DB
▼
  • Apenas SELECT e verificações de schema — somente leitura
  • Suporta PostgreSQL (psycopg2), MySQL (connector), SQLite, SQL Server (pyodbc)
  • Driver instalado antes do teste de conexão — evita ImportError mascarado como "Banco inacessível"
  • Modo real: conecta ao banco via DB_CONNECTION_STRING com credenciais mascaradas no relatório
  • Modo simulado: cria SQLite :memory: com schema inferido dos steps — resultados marcados com simulated: true
🔵 Tipos cobertos: banco

🔄 Ciclo de Vida dos Testes

Setup → Teste → Teardown

Quando testes dependem de dados pré-existentes, os executores seguem um ciclo completo de gestão de massa de dados.

🏗️
Setup
beforeAll
Cria massa de dados via API.
Registra IDs criados.
Se falha → testes vão para skipped.
→
🧪
Execução
Testes rodam contra dados criados no setup.

Falhas são registradas mas não interrompem o teardown.
→
🧹
Teardown
afterAll / finally
Apaga tudo que foi criado.
Sempre executa, mesmo se testes falharem.
Falha no teardown não altera resultado dos testes.
✅ Se setup ok → testes executam normalmente
⏭️ Se setup falha → todos os testes do grupo são skipped com motivo
🧹 Teardown sempre roda, independente do resultado dos testes
⚠️ Se teardown falha → erro registrado, status dos testes não muda

🔑 Resolução de Auth e Credenciais

Prioridade 1 → 4

Todos os executores seguem a mesma ordem de prioridade para obter credenciais antes de iniciar qualquer execução.

1

Token nos steps

O caso de teste já inclui o token ou credencial nos seus steps — nenhuma ação necessária.

✅ Usa direto — sem perguntas
2

Credenciais sem token

Os steps têm usuário/senha mas não token. O executor tenta gerar o token automaticamente via:

POST /auth/login POST /api/login POST /oauth/token POST /token
⚡ Gera token automaticamente
3

Sem credenciais nos steps

Os steps não contêm nenhuma credencial ou token. O executor pergunta ao usuário uma única vez antes de iniciar.

❓ Pergunta ao usuário — UMA vez
4

Usuário não fornece

O usuário foi perguntado mas não forneceu as credenciais solicitadas.

⏭️ Testes skipped com motivo registrado

Casos especiais:

🌐 URL do ambiente: orquestrador infere dos steps; se não conseguir, pergunta ao usuário antes de iniciar.
🗄️ DB_CONNECTION_STRING: executor-banco verifica variável de ambiente; se ausente, pede ao usuário ou skipa automaticamente.

📦 Dependências de Runtime

Pré-requisitos

Alguns binários precisam estar instalados na máquina antes da execução. Dependências Python/Node leves são instaladas automaticamente sob demanda.

🌐 executor-browser / visual / acessibilidade

Node.js + Playwright + binários do Chrome

npm install -D @playwright/test npx playwright install chromium

📈 executor-performance

Binário k6 — não instalável via npm

winget install k6

Se ausente: fallback Python automático

🔌 executor-api / seguranca

Python + biblioteca requests

pip install requests

🗄️ executor-banco

Driver instalado automaticamente por tipo:

pip install -q psycopg2-binary pip install -q mysql-connector-python pip install -q pyodbc

📊 consulta-treinamento

Instalado automaticamente na primeira execução

pip install openpyxl -q

🗃️ Artefatos temporários

Os executores geram arquivos efêmeros na raiz do repositório durante a execução:

tmp_*.py · tmp_*.js · tmp_*.json

Cobertos pelo .gitignore — não são commitados.
